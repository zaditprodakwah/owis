# Adoption Guide

## Purpose

Help developers and organizations transition to using OWIS with minimal friction.

## Migration Paths

| Scenario | Prerequisites | Migration Steps | Risks | Rollback |
|----------|---------------|-----------------|-------|----------|
| **Greenfield** | No existing architecture tooling. | 1. Create a new repository using the **Minimal** template. 2. Add `workspace.json` generated by `owis init`. 3. Run `owis lint` to verify. | Low – fresh codebase. | Delete the repository or revert to initial commit. |
| **Brownfield** | Existing monorepo with custom scripts. | 1. Run `owis init --migrate` to generate a provisional `workspace.json`. 2. Incrementally add `owis lint` checks to CI. 3. Fix violations iteratively. | Medium – may surface hidden architectural violations. | Keep original build scripts; revert `workspace.json` if needed. |
| **Enterprise** | Multiple internal packages, strict governance. | 1. Adopt the **Enterprise Monorepo** template. 2. Align internal policy with OWOS governance (RFC process). 3. Deploy OWIS GitHub Action across all repos. | High – policy adoption required. | Maintain a parallel legacy CI pipeline during transition. |
| **OSS** | Public open‑source library. | 1. Add OWIS as dev dependency. 2. Enable OWIS GitHub Action for PR checks. 3. Publish documentation using VitePress integration. | Low – community contributions must respect `workspace.json`. | Disable action; revert `workspace.json`. |
| **Monorepo** | Multiple inter‑dependent packages. | 1. Use **Enterprise Monorepo** template. 2. Configure `workspace.json` package boundaries. 3. Enforce linting via CI. | Medium – dependency graph complexity. | Revert `workspace.json` or disable lint step. |
| **AI‑first** | Projects primarily driven by LLM agents. | 1. Integrate `owis context` into prompt pipelines. 2. Use the **AI Agent** template to bootstrap. 3. Validate token usage via MCP benchmarks. | Medium – token limits and context freshness. | Switch to static code generation without OWIS context. |

## Prerequisites

- Node.js ≥ 20
- `@prodakwah/owis-cli` installed (`npm i -g @prodakwah/owis-cli`)
- Access to the OWIS GitHub Action (public). 

## Detailed Steps (Greenfield Example)

1. **Bootstrap**:
   ```bash
   npx degit zaditprodakwah/owis-template/minimal my‑project
   cd my‑project
   npm install
   ```
2. **Initialize OWIS**:
   ```bash
   owis init   # generates workspace.json
   ```
3. **Run Lint**:
   ```bash
   owis lint   # should exit 0 for a clean project
   ```
4. **Add CI** (GitHub Action):
   ```yaml
   # .github/workflows/owis.yml
   name: OWIS Checks
   on: [push, pull_request]
   jobs:
     lint:
       runs-on: ubuntu-latest
       steps:
         - uses: actions/checkout@v4
         - name: Setup Node
           uses: actions/setup-node@v4
           with:
             node-version: 20
         - name: Run OWIS Lint
           uses: zaditprodakwah/owis-action@v1
           with:
             command: lint
   ```
5. **Generate Context for AI**:
   ```bash
   owis context > context.md
   # Feed `context.md` to your LLM prompt
   ```

## Risks & Mitigations

- **False Positives**: Early lint rules may be over‑restrictive. Mitigate by tuning `workspace.json` policies.
- **Performance**: Large monorepos can cause long lint runs. Use caching (see the GitHub Action design) and incremental builds.
- **Policy Drift**: Ensure RFCs are updated when architecture changes; otherwise, the lint suite may reject valid changes.

## Rollback Strategy

- Keep the original repository state in a dedicated branch (`pre‑owis`). If adoption fails, switch CI pipelines back to the previous state and delete `workspace.json`.
- The `owis` CLI provides a `owis revert` command (experimental) to restore the last known good configuration.
