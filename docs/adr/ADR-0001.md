# ADR-0001: Core Architecture and Determinism

**Status**: Accepted
**Date**: 2026-07-15

## Context
As the Open Workspace Intelligence Specification (OWIS) transitions from a theoretical specification into a robust reference implementation (v0.2.0-rc.1), several fundamental architectural decisions had to be codified to ensure the longevity, stability, and broad compatibility of the ecosystem.

## Decisions

### 1. Standalone Packages
**Decision**: The OWIS tooling ecosystem is decomposed into distinct, standalone packages (`@prodakwah/owis-runtime`, `-sdk`, `-cli`, `-lint`, `-graph`, `-context`).
**Reasoning**: Decoupling allows ecosystem consumers (like CI/CD pipelines, Editor Extensions, and AI Agents) to consume only the parts they need (e.g., using `runtime` without `cli` overhead) and prevents dependency bloat.

### 2. Additive WIR Graph
**Decision**: The Workspace Intelligence Record (WIR) graph extraction is designed to be purely additive. Nodes (files, modules, packages) and Edges (imports, contains) are extracted with rigid depth and node limits.
**Reasoning**: Prevent deep recursion, cyclic dependency hangs, and out-of-memory (OOM) errors on extraordinarily large monorepos.

### 3. Provider-Neutral Context Layer
**Decision**: The Context Layer uses an adapter pattern (`json`, `markdown`, `openai`, `anthropic`, `langchain`) rather than hardcoding integrations to specific LLMs.
**Reasoning**: Prevents vendor lock-in. It allows OWIS to serve as a pure specification that works elegantly with any upstream LLM workflow tool or framework.

### 4. Deterministic Artifacts
**Decision**: All serialization, iteration, and outputs (`wir.json`, `context.md`, etc.) are aggressively normalized (alphabetical key sorting, normalized POSIX paths, strict UTF-8 boundaries).
**Reasoning**: Byte-identical artifacts are critical. This allows cryptographic hashing (SHA256) of a workspace context, which is essential for caching layers, immutable prompt construction, and rigorous CI test assertions.

### 5. Backward Compatibility
**Decision**: The v0.2.0 runtime is explicitly designed to tolerate and parse v0.1.0 specifications and legacy `workspace.json` payloads without breaking.
**Reasoning**: Ensures early adopters are not punished by ecosystem maturity, aligning with strict Semantic Versioning.

## Consequences
- Modifying core graph logic requires updating cycle detection and memory boundaries.
- Adding a new package requires updating the monorepo tooling and ensuring version lockstep.
- Ecosystem tooling can safely hash OWIS output artifacts and trust they will not change across identical runs.
